This implementation spec operationalizes `intent-001` for remote multi-agent execution.

It focuses on boundaries that must hold even when the live session is no longer generated by a single local runtime:

## Goal
Make remote multi-agent sessions feel like one coherent simulation rather than:
- ordinary chat with roleplay attached later
- a leaked orchestration pipeline
- a well-started meeting that collapses at closing

The first implementation focus should be:
1. transcript boundary integrity
2. closing and evaluator boundary integrity
3. natural but controlled turn ownership

## Problem Statement
Recent playtests suggest the largest naturalness failures in remote multi-agent mode are not primarily wording problems.

They are boundary failures:
- orchestration text becomes visible to the user
- initialization leaks chat-native wrappers before the session actually starts
- facilitator closes the meeting, then evaluator-like or action-driving language appears in the same voice layer
- player-facing turn flow becomes unnatural when actor turns fire before the player can enter or when overlap is handled too rigidly

If these boundaries are not stabilized, conversation quality improvements in persona prompts will not matter much.

## Scope
- In scope:
  - remote multi-agent initialization boundary
  - suppression of orchestration and progress text from user-facing simulation transcript
  - facilitator/evaluator handoff contract
  - closing behavior in remote multi-agent runs
  - turn-design rules for player entry and same-topic overlap
  - evaluator evidence-pack boundary at the architecture level
- Out of scope:
  - rewriting the failure model
  - changing the scene's primary learning goal
  - detailed persona rewriting unrelated to remote orchestration
  - frontend rendering implementation details beyond contract assumptions

## Runtime Boundary Contract

Implementation mapping:
- transcript hygiene should primarily live in `runtime/presentation/`
- evaluator evidence packaging should primarily live in `runtime/transcripts/`
- evaluator judgment should primarily live in local runtime control, not in a remote speaking agent by default
- boundary validation should primarily live in `runtime/validation/` and `runtime/observability/`

### 1. Simulation-Facing Transcript Boundary
The user-facing session transcript must contain only simulation-facing content once initialization begins.

It must not expose:
- orchestration progress text
- tool or worker status text
- "worked for X seconds" style wrappers
- intermediate planning commentary
- actor-generation direction messages

Remote mode should appear to the user as:
- one clean initialization block
- one clean live session
- one clean reflection block

Not as:
- a visible multi-step agent pipeline

### 2. Initialization Boundary
Remote multi-agent mode must preserve the existing simulation initialization contract.

Requirements:
- one clear simulation-facing initialization message
- optional setup clarification if the contract allows it
- no second assistant-style re-explanation before `Start`
- no visible handoff chatter between controller and actors

The user should feel:
- "the simulation is about to start"

not:
- "the system is still assembling the simulation"

### 3. Live Turn Boundary
Once the live session starts:
- only actor/facilitator dialogue should be visible
- there should be no system-side explanation of how the next turn is being generated
- player messages should be followed by actor or facilitator output directly, without visible orchestration filler

### 4. Closing Boundary
Closing belongs to the facilitator.
Reflection belongs to the evaluator.

The remote runtime must keep those layers distinct.

The facilitator may:
- close the meeting
- collect brief participant understanding and cooperation statements
- state that the meeting is ending

The facilitator must not:
- emit evaluator-style scoring language
- issue reflection summaries that read like a post-game report
- assign post-game deliverables in evaluator voice

### 5. Evaluator Boundary
The evaluator begins only after the live session has clearly ended.

Default MVP stance:
- evaluator judgment is local-first
- remote help, if any, should be limited to optional prose shaping after evaluation evidence and judgment are already fixed locally

The evaluator may:
- produce the reflection report
- interpret structural progress
- identify major risks and next questions

The evaluator must not:
- continue in-world meeting dialogue
- assign the player one more final answer after closing
- propose artifact creation unless the user explicitly asks

## Turn Design Contract

### Player Entry Rule
The player must get an early live turn before stakeholder-to-stakeholder discussion becomes substantive.

Avoid:
- facilitator opening -> stakeholder prompt -> stakeholder exchange before player articulation

Preferred:
- facilitator opening
- player initial direction
- stakeholder reaction

### Same-Topic Overlap Rule
Remote multi-agent mode should allow natural stakeholder overlap when it preserves meeting realism.

Allowed pattern:
- stakeholder A
- player
- stakeholder B reacts on the same topic
- player responds

Allow same-topic overlap only when:
- the active topic is still the same
- the second stakeholder concern clearly connects to the immediately preceding exchange
- unresolved burden does not expand beyond playable limits
- facilitator intervention is not needed for legibility

### Anti-Pile-On Rule
Natural overlap is allowed.
Pile-on is not.

The runtime should block or park follow-on turns when:
- multiple unresolved asks accumulate at once
- the player has not yet had a reasonable chance to respond
- actor overlap would cause topic branching rather than same-topic continuation

### Turn Granularity Rule
Remote multi-agent mode should not rely on a naive alternating-turn schedule.

Instead, turn ownership should be selected by:
- active topic
- recent unresolved burden
- same-topic continuity
- whether the player owes a response
- whether facilitator intervention is required

## Evaluation Architecture Contract
The evaluator should not depend on raw full-context replay alone.
It should also not default to remote judgment for MVP.

Preferred architecture:
1. local evidence extraction
2. local attachment of relevant failure slices or symptom candidates
3. local evaluation judgment
4. optional remote prose shaping only if needed

This means the core design problem is:
- where evaluation evidence is stabilized

not simply:
- where evaluation prose is written

MVP recommendation:
- keep evaluator reasoning and scoring local
- only externalize wording polish if the local evaluator has already fixed the judgment

### Evaluator Input Expectation
A future evaluator input packet should likely include:
- key turn evidence
- structural state changes
- parked or unresolved items
- relevant failure slices or candidate symptom ids
- current scene phase and scoring context

## Change Contract
- Allowed Changes:
  - introduce remote-mode runtime boundaries under intent-001
  - define transcript hygiene expectations for multi-agent execution
  - define facilitator/evaluator separation for remote runs
  - define turn-ownership rules needed for natural same-topic overlap
- Forbidden Changes:
  - exposing orchestration internals as part of the visible simulation UX
  - collapsing facilitator and evaluator into one visible voice layer in remote mode
  - allowing actor overlap to degrade into repeated pile-on as a default pattern
- Approval Required:
  - changes that alter the visible session lifecycle itself
  - changes that replace the current reflection format
- Validation:
  - remote runs remain legible as one simulation, not chat plus tooling
  - evaluator output remains phase-aware and separate from closing
  - turn design preserves player entry and same-topic continuity
- Rollback:
  - revert remote boundary logic while keeping the underlying multi-agent foundation if remote UX proves too unstable

## Implementation Outline
1. Define a remote transcript boundary so user-facing simulation output excludes orchestration wrappers and progress messages.
2. Define a clean lifecycle boundary between initialization, live session, closing, and evaluation.
3. Add remote turn-design rules for player entry, same-topic overlap, and anti-pile-on behavior.
4. Define the evaluator evidence-pack concept and where it is assembled.
5. Validate remote sessions against transcript hygiene, closing separation, and turn continuity checks.

## Verification
1. A remote session should start with one clean initialization block and no visible orchestration wrappers.
2. After `Start`, player turns should be followed directly by actor/facilitator content, not temporary system progress text.
3. The player should get an early live turn before stakeholder-to-stakeholder discussion meaningfully advances.
4. Same-topic overlap should be possible without causing multi-topic pile-on.
5. The facilitator should end the meeting without leaking evaluator-style report language.
6. The evaluator should begin only after the meeting clearly ends and should not propose artifact creation unless explicitly asked.
